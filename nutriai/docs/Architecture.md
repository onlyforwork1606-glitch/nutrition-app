# NutriAI — Architecture

NutriAI is a client-first PWA for AI-powered nutrition tracking, backed by a
Cloudflare-native serverless API. The OpenRouter API key lives **only** on the
server — the browser never sees it.

## High-level flow

```
Browser (React PWA)
  │  food photo / coach message
  ▼
Cloudflare Worker  (/api/vision, /api/coach, /api/meals, ...)
  │  - validates session (signed HttpOnly cookie)
  │  - enforces rate limits (KV)
  │  - calls OpenRouter with server-side key
  │  - enriches foods against a curated DB (D1)
  ▼
OpenRouter ──► Gemma / Nemotron / Llama (fallback chain)
D1 (users, goals, meals, coach_messages, ...)
R2 (temporary meal images)
KV (rate-limit counters, OAuth state)
```

## Why server-side AI

- The OpenRouter key is a secret. If it ships in a `VITE_` env var it is
  visible to anyone who opens the bundle.
- Server-side lets us add rate limiting, usage accounting, and a fallback
  model chain without trusting the client.
- CORS + signed cookies keep the API locked to the first-party frontend.

## Components

| Layer        | Tech |
|--------------|------|
| Frontend     | React 18, TypeScript (strict), Vite, Tailwind v4, TanStack Router/Query, Three.js, Framer Motion, Recharts, Zod |
| API          | Cloudflare Worker (Hono), D1, R2, KV, Durable Object, Cron Triggers |
| Auth         | Signed HS256 session cookie; Anonymous by default; Google OAuth optional |
| AI           | OpenRouter: `google/gemma-4-26b-a4b-it:free` (vision + coach), `nvidia/nemotron-nano-12b-v2-vl:free` fallback |
| Storage      | IndexedDB (offline-first local cache) + D1 (server source of truth) |

## Request lifecycle

1. `app.use("*")` assigns a request id, sets security headers, handles CORS.
2. `getOrCreateUser` reads the session cookie; if absent, creates an anonymous
   user in D1 and sets a new cookie.
3. Routes that hit OpenRouter call `rateLimit` (KV fixed windows: 20/min,
   50/day free, 1000/day credited).
4. The Worker calls OpenRouter (falling through the model list on 429/5xx).
5. Vision responses are reconciled against a curated `food_db` so values come
   from real nutrition data, not model hallucination.
6. Responses are logged with request id + latency via structured logs.

## Scheduled jobs

- `0 9 * * 1` → weekly report for every registered (non-anonymous) user.
- `0 9 1 * *` → monthly report.

Reports are generated by the coach service and stored as assistant messages.
